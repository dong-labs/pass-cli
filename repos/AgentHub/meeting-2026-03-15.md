### 2. 架构设计（Arch）

**Arch：** 我建议三层架构：

```
┌─────────────────────────────────────┐
│           CLI Interface             │
│  (agenthub命令，用户交互)             │
├─────────────────────────────────────┤
│           Core Logic                │
│  (搜索、安装、验证、发布)              │
├─────────────────────────────────────┤
│          Adapters                    │
│  (Registry存储, CLI依赖安装)          │
└─────────────────────────────────────┘
```

**技术选型建议：**

| 组件 | 选择 | 理由 |
|------|------|------|
| 语言 | Python | 与现有 OpenClaw 技术栈一致 |
| CLI 框架 | Click / Typer | 成熟、类型安全 |
| 配置 | YAML | agent.yaml 已定义 |
| 存储 | Git Registry | 复用 ClawHub 基础设施 |
| 依赖管理 | pip/npm 安装脚本 | 调用系统包管理器 |

**Registry 设计：**


### 3. 实现方案（CLI Dev + Backend）

**CLI Dev：** 我建议命令结构：

```bash
agenthub <command> [options]

核心命令：
├── search <keyword>     # 搜索智能体
├── info <agent-id>      # 查看详情
├── install <agent-id>   # 安装
├── list                 # 列出已安装
├── update <agent-id>    # 更新
├── uninstall <agent-id> # 卸载
└── init <name>          # 初始化新智能体包

作者命令：
├── validate             # 验证包格式
├── package              # 打包
└── publish              # 发布
```

### 5. 讨论与争议

**争议点 1：Registry 实现方式**

| 方案 | 优点 | 缺点 |
|------|------|------|
| Git Registry | 简单、可复用 ClawHub | 实时性差 |
| 专用 API | 实时性好、功能丰富 | 需额外服务 |

**决议：** v1 使用 Git Registry，后续可扩展 API

**争议点 2：依赖安装安全性**

- **问题：** 自动执行 `pip install` 有安全风险
- **方案：**
  1. 显示要执行的命令，要求用户确认
  2. 支持 `--no-confirm` 参数
  3. 签名验证（未来）

**决议：** 默认确认，支持 `--yes` 参数

**争议点 3：与 OpenClaw 的集成度**

| 方案 | 描述 |
|------|------|
| 独立工具 | 完全独立，任何平台可用 |
| OpenClaw 插件 | 作为 OpenClaw 的一部分 |

**决议：** 两者兼顾 - CLI 独立可用，OpenClaw 内置集成

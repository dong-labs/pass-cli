# 开始新版本规划

分析当前状态，给出版本建议，创建新版本目录。

> **触发关键词**: "start-version", "新版本", "开始版本", "规划版本"

## 执行流程

### 1. 分析当前状态

收集以下信息：
- **当前版本号**：从 `app/build.gradle` 读取
- **上次发版时间**：从最近 tag 获取
- **当前分支**：应该为 develop
- **Git 状态**：是否有未提交更改
- **Issues 状态**：
  - 扫描 `issues/当前版本/` 目录
  - 统计 Todo/In Progress/Done 数量
- **待办 Bugs**：统计 fix/ 类型未完成数量
- **待办功能**：统计 feat/ 类型未完成数量
- **代码变更量**：距上次 tag 的 commits 数

### 2. 智能版本建议

根据分析结果，给出建议：

| 情况 | 建议 | 理由 |
|------|------|------|
| 有 P0/P1 bug | **patch** | 紧急修复优先 |
| 待办 bugs > 3 | **patch** | Bug 修复版本 |
| 有小功能需求 | **minor** | 功能版本 |
| 距上次发版 > 2周 | **patch** | 保持发版节奏 |
| 距上次发版 > 6周 | **minor** | 功能积累够多了 |
| 有重大变更 | **major** | 架构升级等 |
| 无待办事项 | **patch** | 继续小步迭代 |
| 功能需求 > 5个 | **minor** | 功能版本 |

### 4. 用户确认后

创建新版本目录和规划文件：

```bash
✅ 创建目录: issues/232/
✅ 创建版本规划: issues/232/VERSION_PLAN.md

📝 后续步骤:
1. 有 Bug/需求时，使用 /new-task 创建任务
2. 定期使用 /status 查看进度
3. 全部完成后使用 /can-release 检查发版条件
```

## 注意事项

- **当前分支检查**：必须在 develop 分支
- **Git 状态检查**：建议无未提交更改
- **版本目录检查**：如果目录已存在，提示用户
- **兼容性检查**：确保上一个版本的所有 issues 都是 Done（或给出警告）

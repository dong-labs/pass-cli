# 产品愿景

> 让每个人都能轻松创建自己的 AI Bot

---

## 1. 出发点

### 1.1 问题

**国内缺少一个好用的 AI Bot 创建平台**

| 问题 | 说明 |
|------|------|
| **体验差** | 微信/飞书创建 Bot 流程复杂，审核严格 |
| **门槛高** | 需要开发能力，普通人用不了 |
| **AI 集成难** | 想用 AI，需要自己写代码 |
| **渠道分散** | 微信、飞书、企业微信各自独立 |

### 1.2 启发

**Telegram Bot 的体验太好了**

| Telegram 优点 | 说明 |
|---------------|------|
| **创建简单** | 找 @BotFather，一条命令搞定 |
| **立即可用** | 无需审核，秒级上线 |
| **开发者友好** | API 清晰，文档完善 |
| **AI 友好** | 多轮对话、文件、按钮都支持 |

### 1.3 机会

**OpenClaw 正在崛起，需要 Bot 渠道**

- OpenClaw 有强大的 Agent 能力
- 但用户缺少便捷的使用渠道
- 微信/飞书是主要使用场景
- 缺少一个连接 OpenClaw 和 IM 的桥梁

## 2. 产品定位

### 2.1 一句话定位

**OpenClaw Bot 平台 - 国内版 Telegram Bot 体验**

### 2.2 核心价值

```
用户 → App 创建 Bot → 绑定 OpenClaw Agent → 接入微信/飞书 → 立即使用
```

### 2.3 对标产品

| 产品 | 我们的区别 |
|------|------------|
| **Telegram Bot** | 我们面向国内，接入微信/飞书 |
| **微信小程序** | 我们专注 AI Bot，体验更简单 |
| **Poe** | 我们面向 OpenClaw，更开放 |

## 3. 战略选择

### 3.1 核心原则：保持克制，专注 AI

**我们要做什么：**
- ✅ AI Agent 的渠道
- ✅ 让 AI 有机会触达用户

**我们不做什么：**
- ❌ 人跟人对话（微信已经很好了）
- ❌ 通用 IM（大厂竞争，必死）
- ❌ 复杂的社交功能

### 3.2 为什么专注 AI

| 理由 | 说明 |
|------|------|
| **差异化** | 人跟人对话，微信/飞书已经很好了，我们做没有优势 |
| **聚焦** | 做 AI 渠道是空白，做 IM 是红海 |
| **克制** | 功能越少，越容易做好 |
| **验证** | 先验证 AI 渠道需求，再考虑扩展 |

### 3.3 发展路径

```
第一阶段：单 Bot，单用户，单渠道（MVP）
第二阶段：多 Bot，多渠道
第三阶段：多 Agent 协作
第四阶段：人 + AI 混合（如果需要）
```

### 3.4 战略定力

**我们不是要做另一个微信/飞书**

我们是做 **AI 的触角**，让 AI 能够：
- 在微信里和用户对话
- 在飞书里和用户对话
- 在任何渠道和用户对话

**核心竞争力 = AI + 渠道连接能力**

## 6. 未来思考

### 6.1 AI 之间的对话

**场景：Agent A + Agent B 在一个群里协作**

```
用户：@仓咚咚 记一笔账，@思咚咚 分析一下
仓咚咚：已记录，今天花了 50 元
思咚咚：根据分析，你本月餐饮支出偏高...
```

**需要考虑：**

| 功能 | 说明 |
|------|------|
| **群聊支持** | 多个 Agent 在同一个群 |
| **Agent 身份识别** | 区分是哪个 Agent |
| **对话权限** | 谁能调用哪个 Agent |
| **上下文共享** | Agent 之间能否共享信息 |

### 6.2 多 Agent 协作

**场景：用户同时调用多个 Agent**

```
用户：帮我记账并分析
系统：调用 仓咚咚 → 调用 思咚咚 → 聚合结果
```

**需要考虑：**

| 功能 | 说明 |
|------|------|
| **多 Agent 调度** | 如何决定调用顺序 |
| **结果聚合** | 多个 Agent 的结果如何合并 |
| **失败处理** | 部分 Agent 失败怎么办 |

### 6.3 人 + AI 混合群

**场景：工作群里，人和 AI 一起讨论**

```
张三：这个项目进度怎么样了？
AI 助手：根据任务列表，目前完成 60%
李四：那还得抓紧
```

**需要考虑：**

| 功能 | 说明 |
|------|------|
| **触发机制** | AI 什么时候介入（关键词？@？） |
| **权限控制** | 谁能触发 AI |
| **隐私保护** | 敏感话题 AI 不介入 |

### 6.4 权限管理

**谁能调用哪个 Agent？**

| 层级 | 说明 |
|------|------|
| **Bot 所有者** | 创建 Bot 的人，完全控制 |
| **Bot 使用者** | 被授权使用 Bot 的人 |
| **访客** | 临时用户，受限权限 |

**权限模型：**

```
Bot
├── 所有者（Owner）
│   └── 完全控制
├── 成员（Member）
│   └── 可以使用，不能配置
└── 访客（Guest）
    └── 临时使用，有限制
```

### 6.5 计费和用量

**API 调用怎么收费？**

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **按次数** | 每次对话收费 | 低频使用 |
| **按包月** | 固定费用，不限次数 | 高频使用 |
| **按 Token** | 按实际消耗 | 精确计费 |

**用量统计：**

| 数据 | 说明 |
|------|------|
| **调用次数** | Bot 被调用多少次 |
| **Token 消耗** | 实际 API 消耗 |
| **活跃用户** | 有多少人用 |
| **对话时长** | 平均对话长度 |

### 6.6 安全和隐私

**用户数据如何处理？**

| 方面 | 策略 |
|------|------|
| **敏感内容过滤** | 检测并过滤敏感内容 |
| **数据存储** | 对话记录加密存储 |
| **数据保留** | 设置保留期限，到期删除 |
| **隐私政策** | 明确告知用户数据处理方式 |

**安全措施：**

```
1. 传输加密（HTTPS）
2. 存储加密（AES-256）
3. 访问控制（JWT Token）
4. 日志脱敏（敏感信息不记录）
```

### 6.7 监控和运维

**Bot 挂了怎么办？**

| 功能 | 说明 |
|------|------|
| **日志系统** | 记录所有操作和错误 |
| **监控告警** | 实时监控，异常告警 |
| **快速恢复** | 自动重启，快速回滚 |
| **降级策略** | API 失败时的备用方案 |

**监控指标：**

| 指标 | 说明 |
|------|------|
| **可用性** | Bot 在线率 |
| **响应时间** | 平均响应时间 |
| **错误率** | 失败请求占比 |
| **吞吐量** | QPS |
